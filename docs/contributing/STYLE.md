# STYLE.md - SwiftII Coding Style

**Code is read more often than it's written.** Optimise for the human
reading it six months from now, debugging in an emulator with `printf`
and tears - not for the cleverest possible expression. When in doubt,
write the obvious version.

The aesthetic we're aiming for is "professional compiler/interpreter
source that you can still read on a Sunday afternoon." Concrete
references below.

---

## Influences and idioms

The codebases we look to when in doubt - all small, all readable, all
written by people who knew what they were doing:

- **[Crafting Interpreters](https://craftinginterpreters.com/)** (Bob Nystrom) - `clox` is
  the closest match to our shape: single-pass Pratt compiler, stack VM,
  tagged-union `Value`, opcode dispatch on a `switch`. Our `Chunk`,
  `Compiler`, and `VM` structs should feel familiar to anyone who's
  read it.
- **Lua 5.1–5.4 source** - `lopcodes.h`, `lvm.c`, `lparser.c`. The
  reference for clean opcode tables, register VMs, and disciplined
  arena/GC code. Don't copy the register VM (we're a stack VM), but
  copy the *clarity*.
- **[Wren](https://wren.io/) source** - a Lua-flavored language by Nystrom; modern C,
  small footprint, careful with allocation. Good model for a
  scripting-language runtime that has to fit in a small budget.

### Idioms we adopt

- **Tagged-union `Value`** with type tag + payload; predicates
  (`IS_NUMBER(v)`, `AS_STRING(v)`) as macros over the union.
- **Pratt parsing with a parse-rule table** (one row per token: prefix
  fn, infix fn, precedence). Cleaner than recursive-descent for
  expression grammars and easier to extend.
- **Opcode dispatch table** - one C function per opcode (or a tight
  `switch` with one `case` per opcode) plus an `OpInfo` table that
  declares operand widths and stack effects. The table is what
  `tools/host/disasm/` reads, so it can't lie.
- **Bytecode emission helpers** - `emit_byte`, `emit_short`,
  `emit_constant`. Never poke the chunk buffer directly outside
  `src/compiler/emit.c`.
- **Module structs with documented invariants** - `Lexer`, `Compiler`,
  `VM` each own their state; the header comment of `lexer.c`,
  `compiler.c`, `vm.c` spells out what stays true between calls.
- **Short, named locals in hot code** - `tok`, `chunk`, `ip`, `top`
  are fine in the VM dispatch loop. They're idiomatic and the loop
  body is small enough that the reader keeps them in their head.
- **Static-by-default helpers, public API via a clean header** - the
  caller sees only what `lexer.h` chose to expose. Everything else is
  `static` in the `.c` file.

### Idioms we avoid

- **Computed-goto dispatch** (`&&label` extension). Faster on GCC, but
  non-standard and cc65 doesn't support it. Stick to a clean `switch`.
- **Preprocessor abuse** - no `#define BEGIN {` cuteness, no macros
  that expand to multiple statements without `do { ... } while (0)`,
  no opcode handlers generated by `#include` of a list file. The
  opcode list lives in [`OPCODES.md`](OPCODES.md) and the C enum in
  [`src/vm/opcodes.h`](../../src/vm/opcodes.h) is hand-maintained against it.
- **Bit-packing for its own sake**. Pack only when the size budget
  actually demands it, and add a comment pointing at the budget.
- **Single-letter names outside the tightest loops**. `i`/`j` in a
  three-line loop is fine; `c`, `n`, `p` as function-scope locals
  carrying real meaning are not.
- **"Clever" pointer arithmetic** instead of an index + base. The
  index version compiles to the same code on a real C compiler and
  reads in half the time.
- **Hidden control flow** - no `setjmp`/`longjmp`, no goto-as-error-
  handling beyond the textbook single-label cleanup pattern, no
  function pointers buried in structs without a comment naming the
  contract.

When you find yourself reaching for an idiom that isn't on either
list, look at how `clox` or Lua handles the same situation first.
Nine times out of ten they've already picked the answer.

## Language standard

- **Cross-compiled code** (everything under `src/`, except
  `src/platform/host/`): **ISO C90** (also known as ANSI C89) as
  supported by cc65. No C99/C11 features - no mixed declarations, no
  designated initialisers, no compound literals, no variable-length
  arrays, no `//` line comments outside Doxygen markers, no
  fixed-width type literals like `0u`. No GNU or compiler-specific
  extensions. cc65 quirks beyond the spec are captured in
  `feedback_cc65_quirks.md` (auto-memory) and [`LESSONS.md`](LESSONS.md).
- **Host-only code** (`src/platform/host/`, `tests/unit/`,
  `tests/sim/`, `tools/`): **ISO C17** under clang with
  `-std=c17 -Wall -Wextra -Werror -fsanitize=address,undefined`.
  `printf`, `malloc`, and other banned-on-target functions are
  allowed here because these files are never compiled with cc65.

C90 is forced by cc65; C17 on host buys us stronger static checking
and ASAN/UBSAN at test time. Common code that runs in both
configurations must stay within the C90 subset.

## Formatting

- Two-space indent, no tabs.
- Braces on the same line as the controlling statement (K&R style).
- Soft line limit 80 columns; hard limit 100. Wrap long argument
  lists or string literals across lines rather than running past.
- One blank line between top-level definitions; no leading or
  trailing blank lines inside a function body.
- Spaces around binary operators; no space after unary operators or
  inside parentheses.
- One declaration per line in production code (`int x; int y;`, not
  `int x, y;`). Tight test code may relax this.

## Naming

- `snake_case` for functions, variables, struct fields, and file
  names (`parse_expression`, `tok_pos`, `globals.c`).
- `SCREAMING_SNAKE_CASE` for `#define` macros and enum members
  (`OP_HALT`, `FILE_BC_SIZE`, `BUILTIN_PRINT`).
- `CamelCase` for `typedef`'d struct/enum types (`Lexer`,
  `CompileResult`, `Value`).
- File-static identifiers prefixed `s_` to mark module-private state
  at a glance (`s_globals`, `s_count`).
- Module prefix on public functions so the caller sees who owns it
  (`parser_fail`, `globals_define`, `lexer_init`,
  `platform_write_str`).

## Comments

Comment the *why*, not the *what*. The code already shows what; the
comment exists for the things the code can't show - invariants,
hidden dependencies, deliberate trade-offs, workarounds for known
quirks.

**Quantity is not the metric - value is.** "More comments" is not a
quality signal; if anything, the codebases listed under *Influences*
above (`clox`, Lua, Wren) are *sparser* than average, not denser. The
target is a professional compiler/interpreter source: clear naming,
small focused functions, and comments concentrated on the handful of
spots where the reasoning isn't visible in the code. A file with one
sharp comment per non-obvious decision reads better than the same
file with ten that paraphrase the next line.

A useful test: imagine deleting the comment. If a future reader
would be momentarily confused or would re-derive the same conclusion
from scratch (or worse, get it wrong), keep it. If they wouldn't
notice, delete it.

- **Header comment on every `.c` file**: one paragraph describing
  what the module does and the invariants it maintains. Note when a
  feature is missing on purpose (and where the rest lives, e.g. the
  ROADMAP).
- **Doxygen-ish `///` comments** on every non-static function
  declared in a public header - what it does, parameter meaning
  when non-obvious, return value, error modes. Plain `/* */`
  comments are also fine.
- **Inline comments** only when the *why* is non-obvious: a hidden
  invariant, a subtle ordering requirement, a workaround for a
  specific cc65 or Apple II quirk. Reference [`LESSONS.md`](LESSONS.md)
  when the comment exists because of a past bug we recorded -
  future-us appreciates the breadcrumb.
- Do **not** paraphrase code (`// increment i` next to `i++` adds
  noise, not signal). Do **not** narrate the current task in code
  ("added for the FizzBuzz fix", "used by the REPL flow") - that
  belongs in the commit message and rots as the codebase evolves.

### Good vs noise - two pairs

Good (preserved knowledge the code can't show):

```c
/* `?\?/` rather than `??/` to keep clang from interpreting the literal
 * as C's own trigraph for `\` - see -Wtrigraphs. The runtime string is
 * still the three bytes `??/`. */
platform_write_str("  <% -> {   :> -> ]   ?\?/ -> \\\n");
```

```c
/* Order matters: vm_reset_globals releases the global Values' heap
 * refs (so refcounts drop to zero); heap_reset then wipes the heap
 * entirely so any leaked mid-stack allocations are reclaimed too. */
vm_reset_globals();
globals_reset();
heap_reset();
```

Noise (says what the code says):

```c
/* Increment i. */
++i;

/* Loop over all globals and print each one. */
for (i = 0; i < n; ++i) { ... }
```

The good examples answer a question the code raises ("why the
unusual spelling?", "why this order?"). The noisy ones answer no
question at all.

## Functions

- One responsibility per function. If naming it cleanly needs the
  word "and", split it.
- Prefer short functions. Anything past ~80 lines should have a
  good reason to live as one function rather than several.
- Static-by-default: anything not declared in a header should be
  `static`. The linker complains loudly when a helper accidentally
  leaks; better that than a name collision much later.
- Out-parameters for "return a big struct" because cc65 can't
  return aggregates larger than ~4 bytes by value - see
  [`LESSONS.md`](LESSONS.md) "cc65 rejects struct return-by-value".

## Anti-cleverness

A pointer trick that saves three bytes of bytecode is rarely worth
the debugging cost six months from now. We have a 28 KB code budget,
not a 280-byte one - trade size for readability when the size
saving is small. The narrow exceptions are [`CONSTRAINTS.md`](CONSTRAINTS.md)
rule 1 (no float in cross-compiled code) and rule 3 (no printf in
cross-compiled code), which are hard limits and not stylistic.
